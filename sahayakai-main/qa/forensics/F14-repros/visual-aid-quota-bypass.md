# Repro — Image-gen Q4C bypasses daily 10-image quota

**Finding:** F14-P0-2

## Setup
- Teacher T has already generated 9 images today.
- `system_config/feature_flags.visualAidSidecarMode = "canary"`, percent 100 (or teacher is in the bucket).
- `SHADOW_DIFF_IN_CANARY_OBSERVATION = true`.

## Trace

1. T calls `/api/ai/visual-aid` (10th request of the day).
2. `dispatchVisualAid({ userId: 'T' })` → `decision.mode === 'canary'`.
3. `checkImageRateLimit('T')` — counter is 9, limit is 10, NOT exceeded → counter incremented to 10. Pass.
4. `runSidecarSafe(...)` → image #10 (visible) generated by sidecar. **$0.04 billed.**
5. Q4C: `runGenkitSafe(input)` fires in background → image #11 (discarded) generated by Genkit. **$0.04 billed.** **Counter NOT incremented.**

## Result

- T believes they used 10/10 images today.
- Operator billing dashboard shows 10 image-gen calls counted against T.
- **Actual compute consumed:** 11 images.
- **Actual cost charged to project:** $0.44 for T's day (not $0.40 as the dashboard suggests).

## Scale impact

At pilot: 1,000 active teachers/day × 5 visual-aid calls/day each = 5,000 image-gen requests/day.
With Q4C doubling: **5,000 unbilled-against-quota Genkit image-gen calls/day = $200/day uncounted = $6,000/month** hidden from the per-user quota dashboard.

## Same pattern in avatar-dispatch.ts

`src/lib/sidecar/avatar-dispatch.ts` follows the identical structure — primary sidecar call gated by quota, Q4C Genkit observation call ungated.
