# Pydantic Enum Sync — `instant-answer` Wire Contract

**Branch:** `fix/pydantic-enum-instant-answer` (off `develop`)
**Date:** 2026-06-06
**Worktree:** `.claude/worktrees/pydantic-enum-sync/`

## Background

Commit `e119fc10e` (merge of `727522140`) added `'instant-answer'` to two `flow:`
literal unions in `sahayakai-main/src/lib/sidecar/types.generated.ts`
(positions: `LiveToolDefinition.flow`, `VidyaAction.flow`). That file is
auto-generated by `sahayakai-agents/scripts/codegen_ts.py` from the Pydantic
models in the sidecar repo — but only the TS was hand-edited. The Python
source-of-truth Literal types were left unchanged, so any future regeneration
of `types.generated.ts` would silently strip `'instant-answer'` from both
positions and re-break the OmniOrb client (toast: "tool not recognised").

## Changes

### `sahayakai-agents/src/sahayakai_agents/agents/vidya/schemas.py`
- Added `'instant-answer'` to `AllowedFlow` Literal (the wire enum for
  `VidyaAction.flow`).
- Replaced the stale comment ("`instantAnswer` is intentionally NOT in this
  Literal") with a precise explanation of the distinction between
  `AllowedFlow` (wire enum, now 10 values including `instant-answer`) and
  `prompts.ALLOWED_FLOWS` (intent-classifier's routable set, still 9 values,
  uses camelCase `instantAnswer` for inline handling).

### `sahayakai-agents/src/sahayakai_agents/agents/vidya_voice/schemas.py`
- Added `'instant-answer'` to `LiveAllowedFlow` Literal so
  `LiveToolDefinition.flow` regenerates with the same set. The voice tool
  surface itself does not register an instant-answer tool today, but keeping
  the two `flow` enums aligned avoids drift between
  `LiveToolDefinition` and `VidyaAction` consumers of the generated TS.

### `sahayakai-agents/tests/unit/test_vidya_schemas.py`
- Extended `test_each_allowed_flow_accepts` to cover `instant-answer`.
- Added `test_instant_answer_is_in_allowed_flow_literal` — fast-fail guard
  on the Python side.

### `sahayakai-agents/tests/unit/test_action_flow_ts_parity.py` (new)
- Cross-stack CI guard. Reads `sahayakai-main/src/lib/sidecar/types.generated.ts`,
  parses every `flow:` literal union, asserts:
  1. `'instant-answer'` is in both Python Literals (`AllowedFlow`, `LiveAllowedFlow`).
  2. `'instant-answer'` appears in every TS `flow:` literal position (today: 2).
  3. **General drift guard:** every Python `AllowedFlow` value must appear in
     every TS `flow:` union (catches future cases beyond instant-answer).

## What I did NOT change

- `prompts.ALLOWED_FLOWS` list, `registry.SUB_AGENTS`, `_TOOL_DEFINITIONS` in
  `vidya_voice/agent.py`. These are orchestrator/tool-definition catalogs
  and use a different concept (the 9 navigable flows the classifier picks).
  `classify_action` still routes `instantAnswer` (camelCase) inline before
  ever consulting `ALLOWED_FLOWS`, so this is the correct split.
- `len(ALLOWED_FLOWS) == 9` assertion in `test_vidya_agent.py` stays green.
- The hand-edited TS file in `sahayakai-main` was NOT mass-regenerated.
  Running `python scripts/codegen_ts.py` produced a ~265-line diff because
  Python sources have drifted significantly ahead of the committed TS
  (new schemas like `AssessAssignmentRequest`, `AssessmentScannerResponse`),
  AND the codegen has a bug emitting invalid identifiers
  (`RubricSnapshot-Input`, `RubricSnapshot-Output` — hyphens in TS identifier
  names). Both issues are pre-existing and out of scope for this fix; a
  separate task was spawned to address the codegen drift + identifier bug.
